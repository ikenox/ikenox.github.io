継承と委譲の使い分けについて、いざ「どういった基準で使い分けているか？」と聞かれると意外ときちんと言語化できなかったので、この記事に書き起こして言語化を試みました。
なお、「継承 委譲 違い」とかで検索すると先人たちの説明がたくさん出てきますが、それらを自分なりに消化して自分の説明での言語化してみたいと思います。

## TL;DR

自分なりにたどりついた結論を先に述べると、正しい使い分けができている場合には以下のような傾向が存在すると考えます。

- 委譲は子クラスが親クラスより上位の概念になる
- 継承は親クラスが子クラスと同じ抽象度の概念になる
- インターフェースは親クラスが子クラスより上位の概念になる

## 委譲は子クラスが親クラスより上位の概念になる

たとえばSQLデータベースからユーザー情報を取り出したり保存したりする機能を作ろうとしたとき、ぱっと思いつく方針としては、

- DBの接続を保持したりSQLクエリを実行したりする部分のロジックは、ユーザー情報に限らず他の情報を取得するときにも使いまわせそう
- 使いまわせそうな部分は別クラスに切り出しておいて、ユーザー情報の操作専用のクラスからそのクラスを**なんらかの方法**で利用するようにすれば共通ロジックを使いまわせそう
    - なんらかの方法 = 継承 or 委譲

という感じになるかと思います。

結論から言うと委譲を使ったほうが良いのですが、これをもし継承でやると、たとえば以下のようになります。

#### 委譲を使うべき場面で継承を使った例

```java
class Database {
    // 必要に応じてDBへの接続ロジックなどもこのクラス内に書かれる

    Rowdata doSQL(String sql){
        // DBにクエリを投げ、その結果を返すロジック
    }
}
```
```java
class UserDatabase extends Database {
    User getUser(int userId) {
        String sql = String.format("SELECT * FROM users WHERE user_id=%d", userId);
        Result result = this.doSQL(sql)
        // クエリの実行結果をもとにユーザーオブジェクトを作って返す
    }
}
```

DBへの接続やSQLの実際の実行などの共通部分は`Database`が受け持ち、ユーザー情報の操作に関しては`UserDatabase`が受け持つようにはできています。

しかし、この実装にはいくつか歪みが生じてしまいます。

なぜこのような不都合が起きてしまったかというと、

UserDatabaseは「ユーザーを取り出す」という操作を役割として持つのに対して、Databaseは「SQLデータベースに接続したり問い合わせたりする」という、UserDatabaseと比べるとより具体的で技術的な役割を持っています。

`UserDatabase`は、`Database`クラスの「SQLデータベースに接続したり問い合わせたりする」機能を単に利用することによって、より抽象的な「ユーザーを取り出す」という操作を実現したいのであって、`Database`自身になりたいわけではありません。  
極端な例で言うと、「車クラスが、ライトを点けるためにライトクラスを継承する」というのと同じことをやっていると言えます。
逆に言うと、普段

親と子に期待される役割は同じか？を考え、同じでないなら単なるツールとして委譲使うほうが良いといえます。

## インターフェースは親クラスが子クラスより抽象的になる

インターフェースでは、「何ができるか」を表した親（インターフェース）に対して、その親を継承した子クラスが「具体的にどうやるか」を実装します。

このとき、指定されたユーザー情報はアプリによってはSQLデータベースに存在するかもしれないですし、もしかすると`.txt`形式でファイルに保存されているかもしれません。しかし、こちらとしてはそこらへんはどうでも良く、とにかく意図したユーザーのオブジェクトが手に入ればOKです。
つまり、`userRepository`に期待されているのは「特定の本を取得する」という動作であり、`userRepository`を使う側にとって「どう取得するか」というのは全く興味がないといえます。

こういう場合には

```java
interface UserRepository {
    User getUserById(int userId);
    void saveUser(User user);
}
```
```java
class MySQLDatasource implements Datasource {
    User getUserById(int userId){
        // SQLクエリを叩いてユーザーのデータを取得、オブジェクトにして返す
    };
    void saveUser(User user){
        // SQLクエリを叩いてユーザーのデータを保存
    };
}
```
```java
class TextFileDatasource implements Datasource {
    User getUserById(int userId){
        // ユーザーがリスト形式で保存されているtxtファイルを開いてユーザーのデータを取得、オブジェクトにして返す
    };
    void saveUser(User user){
        // ユーザーがリスト形式で保存されているtxtファイルを開いてユーザーのデータを保存
    };
}
```

## 委譲

- レイヤが違う = 責務が違う
- 親が下位クラスということは、子からすると親は*単なるツール*である可能性が高い
- 親クラスが子クラスより下位レイヤの場合、継承を使ってしまうと小クラスはA,Bどちらの機能も同列に管理することとなり、単一責務の原則に反する
- 委譲するとすっきりする、異なる責務
